🔄 Many users coming from competitor tools expected a smooth, continuous scrollable calendar — the month-by-month navigation felt unfamiliar and slow.
The Journey — 2–3 Quarters in the Making
🚀 Ravi kicked it off — initially experimented with building a fully custom calendar from scratch, bypassing FullCalendar entirely.
🧪 First POC used a virtualizer — only ~4 months rendered at a time. Screen jittered during fast scrolling, white-space handling was tricky, and scrolling to a non-rendered month was complex since it didn't exist in the DOM.
🔄 Pivoted to IntersectionObserver — switched to rendering all months in the DOM but lazily activating events via an observer, which eliminated the jitter and gave us smooth, predictable scrolling.

How We Implemented It — 4 Core Pieces
📜

1 · Vertical Scroll Architecture

  • All months rendered as stacked DOM elements inside a single scrollable div.
  • Removed Next/Previous buttons; the scroll container is the navigation.
👁️

2 · Lazy Event Rendering (IntersectionObserver)

  • Rendering events for all 12–24 months at once was too heavy on the browser.
  • We use an IntersectionObserver with a 200 px buffer around the viewport — only nearby months load their events; the rest stay as lightweight shells.
  • Visibility changes are debounced 80 ms to prevent unnecessary re-renders during fast scroll.
🖱️

3 · Cross-Month Drag Selection

  • Drag across month boundaries to select a date range — highlights update in real time.
  • Mouse events tracked on the whole document, so fast movement and scrolling mid-drag never break the selection.
⚡

4 · Auto-Scroll & Performance

  • When dragging near the top or bottom edge of the calendar, it auto-scrolls smoothly — faster the closer you are to the edge.
  • Highlight updates are batched to run at most once per screen refresh, keeping the UI smooth even during fast drags.
  • If the selected date range hasn't changed, highlight work is skipped entirely — no wasted CPU.
